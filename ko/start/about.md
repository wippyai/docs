# Wippy 소개

Wippy는 실행 중에 변경이 필요한 소프트웨어를 위한 에이전틱 플랫폼 및 런타임입니다. 자동화 시스템, AI 에이전트, 플러그인 아키텍처 등 코어가 한 번 엔지니어링되고 재빌드나 재배포 없이 반복적으로 적응해야 하는 애플리케이션에 적합합니다.

액터 모델이 기반입니다. 코드는 메시지를 통해 통신하는 격리된 프로세스에서 실행되며, 각각 자체 상태를 관리합니다. 무언가 실패하면 격리된 상태로 실패합니다. 슈퍼비전 트리가 크래시 시 프로세스를 자동으로 재시작하여 복구를 처리합니다.

```lua
local worker = process.spawn("app.workers:handler", "app:processes")
process.send(worker, "task", {id = 1, data = payload})
process.monitor(worker)
```

설정은 변경 사항을 이벤트로 전파하는 중앙 레지스트리에 있습니다. 설정 파일을 업데이트하면 실행 중인 프로세스가 변경 사항을 수신합니다. 시스템이 계속 실행되는 동안 재시작 없이 적응합니다. 새 연결, 업데이트된 동작 등 필요한 모든 것이 가능합니다.

```lua
local db = registry.get("app.db:postgres")
local cache = registry.get("app.cache:redis")
```

인프라 장애에서 반드시 살아남아야 하는 작업(결제 흐름, 다단계 워크플로우, 장기 실행 에이전트 태스크)의 경우, 런타임이 자동으로 상태를 영속화합니다. 작업 중 서버가 죽어도 워크플로우는 다른 머신에서 멈춘 곳에서 바로 재개됩니다.

전체 시스템이 단일 파일에서 실행됩니다. 오케스트레이션할 컨테이너도, 조율할 서비스도 없습니다. 하나의 바이너리, 하나의 설정으로 런타임이 나머지를 처리합니다.

## 배경

액터 모델은 1980년대부터 전화 교환기를 운영해온 Erlang에서 왔습니다. "let it crash" 철학(실패를 격리하고, 빠르게 재시작)도 거기서 왔습니다. Go는 채널과 메시지 전달이 동시성 코드를 읽기 쉽게 만들 수 있음을 보여주었습니다. Temporal은 내구성 있는 워크플로우가 프레임워크와 싸우지 않아도 됨을 증명했습니다.

우리는 AI 에이전트가 실행 중에 변경할 수 있는 인프라가 필요하기 때문에 Wippy를 만들었습니다. 새로운 도구, 업데이트된 프롬프트, 다른 모델—이것들은 배포 주기를 기다릴 수 없습니다. 에이전트가 새로운 접근 방식을 시도해야 할 때, 그 변경은 릴리스 후가 아니라 몇 초 안에 작동해야 합니다.

에이전트가 레지스트리 접근 권한을 가진 액터로 실행되기 때문에, 코드를 생성하고, 새 컴포넌트를 등록하고, 자체 워크플로우를 조정하는 등 이러한 변경을 스스로 수행할 수 있습니다. 충분한 권한이 주어지면, 에이전트는 인간의 개입 없이 작동 방식을 개선할 수 있습니다. 시스템이 스스로를 작성할 수 있습니다.
